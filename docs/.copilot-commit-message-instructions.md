## Subject Rules

1. Use imperative mood: "add feature" not "added feature"
2. Don't capitalize first letter
3. No period at the end
4. Maximum 50 characters
5. Be specific and descriptive

## Body (optional)

- Explain WHAT and WHY, not HOW
- Wrap at 72 characters
- Separate from subject with blank line

## Footer (optional)

- Reference issues: `Closes #123`, `Fixes #456`
- Breaking changes: `BREAKING CHANGE: description`

## Additional Guidelines

1. **Atomic commits**: One logical change per commit
2. **Test before commit**: Ensure code works
3. **No WIP commits** in main/dev branches
4. **Sign commits** when possible
5. **Reference issues** when applicable

## Breaking Changes

If introducing breaking changes, add to footer:

```
BREAKING CHANGE: changed UpdateClient DTO structure

Backend now expects direct DTO fields instead of nested data object.
Frontend must send { firstName, lastName, email } instead of 
{ data: { firstName, lastName, email } }
```

## Project-Specific Patterns

### Feature Development

- Use descriptive branch names: `4-permitir-al-administrador-modificar-los-datos-del-cliente`
- Reference issue numbers in commits when available
- Group related changes in the same commit

### Backend Development (NestJS)

- Document Prisma schema changes in migration commits
- Update DTOs when modifying request/response structures
- Add proper error handling and logging
- Use proper HTTP status codes and exception filters

### API Changes

- Always document breaking changes in API contracts
- Update TypeScript interfaces when modifying DTOs
- Add validation with class-validator decorators
- Document endpoints with Swagger/OpenAPI decorators

### Database Changes

- Create descriptive migration names
- Never modify existing migrations
- Test migrations locally before committing
- Document schema changes in commit body

### File Upload & Storage

- Mention S3/Wasabi bucket changes
- Document presigned URL generation changes
- Note file size or format restrictions

### Email & Notifications

- Mention template changes (react-email)
- Document email triggers and events
- Note changes to notification tokens
